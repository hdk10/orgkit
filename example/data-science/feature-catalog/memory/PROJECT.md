# feature-catalog — Project Memory

_Last updated: 2026-06-18_

## What it is

The canonical inventory of every feature available for modeling from our data partners. A single `master.csv` with ~300 features across 4 partners, with canonical names, categories, derivation logic, and per-partner runnable ETL transforms. Any DS project building a model should start here.

## Status

- `master.csv` v1.3 locked (hand-curated; do not auto-regenerate).
- Transforms working for all 4 partners.
- `--universe` flag enforced on all transforms (v1.2 fix).
- Open: automated schema-drift mapping for partner extracts with renamed columns.

## master.csv schema

```
id, Partner, Source Variable, Category, Canonical Name, Description, Derivation Logic
```

- Filter by `Partner` to see a given feed.
- Use `Canonical Name` in all model code. Never use `Source Variable` directly.
- 13 categories: demographic, financial, transactional, device, behavioral, telecom, social, credit, risk, digital, b2b, identity, derived.

## Running a partner transform

```bash
# Required: pass the full canonical seed as --universe
python3 by_partner/vendor1_transform.py \
  --input raw/vendor1_extract.csv \
  --universe canonical_seed.csv \
  --output output/vendor1_features.parquet
```

The `--universe` flag is **required**. Without it, absent users are silently dropped instead of flagged with `vendor1_user_not_found = 1`. This was a real bug that caused under-reporting of coverage.

## Adding a new partner

1. Append rows to `master.csv` for each new feature (hand-edit; do not re-run `build_catalog.py` to generate — it would clobber manual curation).
2. Create `by_partner/<partner>_transform.py` following the `DERIVED`/`PASSTHROUGH`/`MATCH_ID` pattern in existing transforms.
3. Add a client entry to `tests/conftest.py CLIENTS` registry for integration tests.
4. Run `python3 build_catalog.py --validate-only` to check referential integrity.

## Adding a derived coverage flag

1. Append a master.csv row with `Category=derived`, `Derivation Logic="user not found in <partner> extract"`.
2. Add `DERIVED = {"canonical_user_not_found": "join-key source column"}` to the transform.
3. Fold into `FEATURE_NAMES`.
4. Add compute line in the transform.
5. `build_catalog.py --validate-only` auto-validates the mapping.

## Known issues / open

- Partner schema drift: some client extracts rename columns (e.g., `metric_name_v2` vs `metric_name`). Currently handled manually per client. Should be automated via a mapping config YAML.
- Enrich partner (Vendor3) uses `email` as the join key, not phone. SHA-256 phone hashes from this partner use a different convention with near-zero overlap with our canonical phone hash. Always verify join key and coverage before feature engineering.

## Lessons captured inline

[GOTCHA]: `master.csv` is hand-maintained and must NEVER be auto-regenerated by re-running `build_catalog.py`. The original build script was kept for historical reference only. Re-running it would clobber ~40 rows of manually curated descriptions and derivation notes. The script now runs in validate-only mode.

[LESSON]: One canonical seed file serves every partner transform. Each transform declares its `MATCH_ID` column (e.g., phone hash or device ID). A partial seed (missing any of the 4 ID columns) is rejected with `SystemExit` at startup. This prevents silent partial joins.

[PATTERN]: `DERIVED` dict pattern for passthrough-derived flags — to add a coverage indicator: (1) append master row, (2) add `DERIVED = {"canonical_name": "source_column"}` to transform, (3) fold into `FEATURE_NAMES`, (4) add compute line. `build_catalog.py check_passthrough_mapping` auto-covers it in validation.

[GOTCHA]: pandas with pyarrow-backed dtypes breaks `.value_counts(normalize=True).values ** 2` and `sum()` over boolean Series. Use `.to_numpy(dtype=float)` / explicit int casts in transform scripts. Hit in Vendor2 transform during PSI computation.
