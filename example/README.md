# Example Org

This directory is a fully worked example showing what a populated orgkit org looks like after a few months of real use. Use it as a **reading guide and copy-paste starting point** — your real org will look like this once it's populated.

---

## What's in here

```
example/
  .org/
    roles.json          ← role → folder mapping (the registry)
  CLAUDE.md             ← global brain: who/what/how (always injected)
  ORG.md                ← auto-generated org map (regenerated by sync_org.py)
  engineering/
    memory/ROLE.md      ← the engineering role brain
    api-service/
      memory/PROJECT.md ← project-level memory with inline [LESSON]/[GOTCHA] tags
    admin-dashboard/
      memory/PROJECT.md
  data-science/
    memory/ROLE.md      ← the data-science role brain (richest example)
    churn-model/
      memory/PROJECT.md
    feature-catalog/
      memory/PROJECT.md
  design/
    memory/ROLE.md
    pitch-deck/
      memory/PROJECT.md
  growth/
    memory/ROLE.md      ← lighter example (smaller role)
```

---

## How to read it

**Start with `.org/roles.json`** — it defines the 4 roles and their project folders. This is the registry `sync_org.py` uses to generate `ORG.md`. When you add a new project, you add a folder here first.

**`CLAUDE.md`** is the global brain. It is injected into every session, regardless of which folder you open in. Keep it short: who the company is, the tech north star, and the cross-cutting rules.

**`ORG.md`** is auto-generated by `sync_org.py` (run by the SessionStart and Stop hooks). Do not edit it manually.

**`<role>/memory/ROLE.md`** is the role brain. It is injected when you open a session inside that role's folder. This is where accumulated craft knowledge lives — best practices, patterns, anti-patterns, gotchas, tools, glossary, solved problems index. The `data-science` and `engineering` ones are the richest examples.

**`<role>/<project>/memory/PROJECT.md`** is project-level memory. Shorter and more specific. Notice the inline `[LESSON]: ...`, `[GOTCHA]: ...`, `[PATTERN]: ...` tags at the bottom — these are scraped automatically by the Stop hook and promoted to the role brain over time.

---

## The capture loop

```
You write  [LESSON]: insight here
     ↓
Stop hook scrapes it from any PROJECT.md
     ↓
_pending.md accumulates raw captures
     ↓
/role-promote <role>  reconciles pending into ROLE.md (deduplicated, structured)
```

You never have to manually update ROLE.md during a session. Just drop `[LESSON]:` / `[GOTCHA]:` / `[PATTERN]:` / `[TOOL]:` tags as you work, and the system promotes them.

---

## Your real org will look like this

When you run `orgkit init` on your own repo, you get the scaffolding. After a few weeks of tagging lessons as you work:

- Each ROLE.md will fill up with the patterns that actually matter in your context.
- Each PROJECT.md will have specific gotchas and open questions for that project.
- CLAUDE.md will hold the things that are true everywhere.

The example org uses a fictional SaaS company ("Acme") so you can read it without any prior context. The structure, depth, and tag style are exactly what a real org looks like.
